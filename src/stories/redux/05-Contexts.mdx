# Redux — Contexts

There are **5 hook modules** in `src/app/context/`. Only `AuthContext` is a real React Context (with `createContext` + `<AuthProvider>`). `CartContext`, `WishlistContext`, `QuickViewContext` are **hook-only modules** — no `createContext`; the hooks call Redux + Server Actions directly. `CatalogAccentContext` is a plain string context provided inline by the catalog pages.

**Why the context layer exists:** components import from these hooks, not Redux slices or Server Actions directly. If the underlying implementation changes (which happened when cart/wishlist sync migrated from RTK Query to Server Actions), only the hook file needs updating — call sites stay untouched.

## CartContext · `useCart()`

Wraps `cartSlice` + `syncCartAction`. Import: `import { useCart } from '../context/CartContext';`

```ts
interface CartItem {
  id: string;              // playground SKU ('wc-1') OR stringified OE product id
  name: string;
  brand: string;
  color: string;
  sku: string;
  size: string;
  quantity: number;
  price: number;
  originalPrice?: number;  // for discount display
  image: string;
  bundleId?: string;       // groups bundle items for removeBundle / bundle quantity
}

interface CartContextType {
  items: CartItem[];
  totalItems: number;
  subtotal: number;
  discount: number;
  total: number;
  miniCartOpen: boolean;
  addItem: (item: CartItem) => void;
  addBundle: (items: Omit<CartItem, 'bundleId'>[]) => void;
  removeItem: (id: string) => void;
  removeBundle: (bundleId: string) => void;
  updateQuantity: (id: string, delta: number) => void;
  updateSize: (id: string, size: string) => void;
  clearCart: () => void;
  openMiniCart: () => void;
  closeMiniCart: () => void;
}
```

**Deduplication:** `addItem` matches by **`id + size`** — same product in different sizes = separate items.

**Server sync:** whenever `items` changes AND `isLoggedIn` is `true`, a debounced 400 ms effect maps items to `OeCartItem[]` via `getCmsProductId(playgroundId)` and calls `useAuth().syncCart(oeItems)`. Failures are silent (fire-and-forget); the next mutation re-attempts.

**Login-time merge:** on the first login after guest browsing (guarded by `sessionStorage['oe_cart_merged']`), the effect fetches placeholders from `user.cartItems`, enriches them via `getProductsByIdsAction`, and merges into Redux. Stale placeholder items (products deleted on OE side) are dropped.

**Bundles:** `updateQuantity` on a bundle line updates **all siblings sharing the same `bundleId`** by the same delta — a single `QtyControl` drives the whole bundle. `syncCart` flattens the bundle into independent `{productId, qty}` entries (OE has no bundle concept); re-hydrating after re-login breaks the bundle back into separate lines.

## WishlistContext · `useWishlist()`

Wraps `wishlistSlice` + `syncWishlistAction`. Import: `import { useWishlist } from '../context/WishlistContext';`

```ts
interface WishlistItem {
  id: string;
  name: string;
  brand: string;
  price: number;
  salePrice?: number;
  image: string;
  colorImages?: string[];
  colors?: string[];
  colorStock?: number[];   // per-color stock counts (same index as colors)
  sizes?: string[];
  badge?: string;
  inStock: boolean;
  priceAlert?: boolean;
  selectedColor?: string;
  selectedSize?: string;
}

interface WishlistContextType {
  items: WishlistItem[];
  count: number;
  isWishlisted: (id: string) => boolean;
  addItem: (item: WishlistItem) => void;
  removeItem: (id: string) => void;
  toggleItem: (item: WishlistItem) => void;
  updateSelection: (id: string, selectedColor?: string, selectedSize?: string) => void;
  clearAll: () => void;
}
```

Same debounced 400 ms sync pattern as `CartContext`, gated on `oe_wishlist_merged` sessionStorage flag.

**`toggleItem` vs `addItem`:** use `toggleItem` for heart buttons (add if absent, remove if present); use `addItem` when you need to guarantee the item is in the list regardless.

**`updateSelection`:** call this when the user changes the selected colour/size on an already-wishlisted item so the change persists to localStorage and syncs to OE.

## QuickViewContext · `useQuickView()`

Hook-only — no `createContext`. Wraps `uiSlice.quickView`. Import: `import { useQuickView } from '../context/QuickViewContext';`

```ts
interface QuickViewContextValue {
  isOpen: boolean;
  product: Product | null;
  initialColorIndex: number | null;
  openQuickView: (product: Product, initialColorIndex?: number) => void;
  closeQuickView: () => void;
}
```

**`initialColorIndex`:** when the user selects a colour in the catalog card and opens Quick View, the selected index is passed through so the modal opens with the correct swatch pre-selected. `ProductCard` passes its `selectedColor` state; other callers (`FavoritesPage`) can omit it.

**"View Full Details" navigation:** clicking "View Full Details" in the modal navigates to `/product/[id]?color=#hex&size=…` with the current variant — `ProductDetailPage` reads it from `searchParams`.

**Close animation:** `closeQuickView()` dispatches `closeQuickView` immediately (sets `isOpen: false`), then calls `clearQuickViewProduct()` after a **300 ms timeout** so the exit animation completes before the product is dropped from state.

## AuthContext · `useAuth()`

Real React Context with `<AuthProvider>` (mounted globally in `Providers.tsx`). Bootstraps the user session from `getCurrentUserAction()` on mount; every mutation is a Server Action from `src/lib/oneentry/auth/actions.ts`.

Import: `import { useAuth } from '../context/AuthContext';`

```ts
interface AuthContextType {
  isLoggedIn: boolean;
  user: User | null;               // merged OeUser + loyalty defaults
  /** false until the bootstrap /me call finishes — gate flash-of-signed-out UI on this */
  authReady: boolean;

  // Modal state
  loginModalOpen: boolean;
  registerModalOpen: boolean;
  openLoginModal: () => void;
  closeLoginModal: () => void;
  openRegisterModal: () => void;
  closeRegisterModal: () => void;

  // Sign-in / sign-up / logout
  login:  (emailOrPhone: string, password: string) => Promise<boolean>;
  loginWithGoogle: (accessToken: string) => Promise<{ ok: boolean; error?: string }>;
  signUp: (input: SignUpInput)          => Promise<{ ok: boolean; error?: string }>;
  logout: () => void;

  // Mutations — all persist through Server Actions
  updateUser:        (data: Partial<User>) => void;                                    // optimistic local merge
  updateProfile:     (patch: ProfileUpdate)             => Promise<{ ok: boolean; error?: string }>;
  updateAddresses:   (addresses: OeAddress[])           => Promise<{ ok: boolean; error?: string }>;
  updateSubscriptions:(subs: OeSubscriptions)           => Promise<{ ok: boolean; error?: string }>;
  updateConsent:     (consent: OeConsent)               => Promise<{ ok: boolean; error?: string }>;

  // Fire-and-forget sync (called from CartContext / WishlistContext)
  syncCart:     (items: OeCartItem[])     => Promise<void>;
  syncWishlist: (items: OeWishlistItem[]) => Promise<void>;
}
```

**Session model:** the storefront no longer keeps JWTs in Redux or `localStorage`. `oe_access` / `oe_refresh` / `oe_user` are httpOnly cookies set by `signInAction` and cleared by `signOutAction`. Every authenticated Server Action reads the cookie server-side; the client only holds the resulting `user` object.

**Bootstrap:** on mount, `AuthProvider` calls `getCurrentUserAction()`. If a valid session cookie exists, the action returns `OeUser` (profile + addresses + subscriptions + consent + cartItems + wishlistItems + oeOrders + recentlyViewedItems). Either way, `authReady` flips to `true`.

**Sign-in flow (login + Redux side-effects):**

```
useAuth().login(emailOrPhone, password)
        │
        ▼
signInAction (Server Action)
        │
        ├─→ getApi().AuthProvider.auth('email', {authData:[{marker:login},{marker:password}]})
        │       │  on success — Server sets oe_access / oe_refresh / oe_user cookies
        │       │
        │       └─→ getCurrentUserAction() (internal) → returns OeUser
        │
        ▼
{ ok:true, userIdentifier, user }
        │
        ├─→ dispatch(setAuth({ accessToken:'', refreshToken:'', userIdentifier }))  // Redux — identifier only
        ├─→ setUser(mergeOeUser(user))                                              // local React state
        └─→ setIsLoggedIn(true) → cart/wishlist sync effects unlock
```

**Google OAuth:** `loginWithGoogle(accessToken)` — the client gets the token from GIS via `requestGoogleAccessToken()` in `src/lib/google-auth.ts` (requires `NEXT_PUBLIC_GOOGLE_CLIENT_ID`), then `signInWithGoogleAction` exchanges it for an OE session.

**Sign-up:** `signUp(input)` — `input` follows the CMS attribute set `users_sign_in_sign_up` loaded via `loadSignUpFormSchema()` and exposed by `SignUpFormSchemaProvider`.

**Forgot password:** the "Forgot password?" link inside `LoginModal` currently fires `alert(L.forgotConfirm)` — a placeholder. There is no `requestPasswordResetAction` yet.

**Logout:** clears local state immediately (optimistic), dispatches `clearAuth()`, then fires `signOutAction()` in the background to clear the cookies and invalidate the session server-side.

**WishlistSyncEffect** (inside `Providers.tsx`): **no-op placeholder** — the real wishlist hydration now runs inside `WishlistContext` from `user.wishlistItems`. The component is retained only for backwards compatibility with the earlier pattern.

## CatalogAccentContext · `useCatalogAccent()`

A simple string context providing the accent hex colour for catalog pages.

Import: `import { useCatalogAccent } from '../context/CatalogAccentContext';`

```ts
const accent = useCatalogAccent();  // '#F88A8A' or '#DA1E1E'
```

Override in a story:

```tsx
export const MensVersion: Story = {
  render: () => (
    <CatalogAccentContext.Provider value="#DA1E1E">
      <ProductCard product={MOCK_PRODUCT} />
    </CatalogAccentContext.Provider>
  ),
};
```
