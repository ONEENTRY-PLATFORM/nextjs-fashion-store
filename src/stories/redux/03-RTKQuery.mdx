# Redux — RTK Query

There are **2 RTK Query APIs** in the store: `cartApi` and `wishlistApi`. Both use `fetchBaseQuery` against `NEXT_PUBLIC_API_URL` with a Bearer header pulled from `state.user.data.authToken`.

**On the current live path these hooks are NOT called.** Cart / wishlist sync goes through Server Actions (`syncCartAction` / `syncWishlistAction`) that read the httpOnly `oe_access` cookie server-side. The RTK Query slices remain registered as compiled scaffolding so a future rollback to client-side sync would only need `NEXT_PUBLIC_API_URL` and hook calls — no store surgery required.

Three older APIs — `productsApi`, `homepageApi`, `catalogConfigApi` — have been **removed**. All product / homepage / catalog config data now comes from OneEntry via RSC loaders in `src/lib/oneentry/**`. See `01-Overview` and `06-ServerActions`.

## Where the live sync path actually lives

Cart and wishlist mutations DO reach OneEntry — just not through these hooks. The path is:

1. Component calls `useCart().addItem(...)` / `useWishlist().toggleItem(...)`.
2. The hook dispatches optimistically to `cartSlice` / `wishlistSlice`.
3. A debounced 400 ms effect in `CartContext` / `WishlistContext` maps the local items to `OeCartItem[]` / `OeWishlistItem[]` via `getCmsProductId(playgroundId)` and calls `syncCart(...)` / `syncWishlist(...)` on `AuthContext`.
4. Those thin wrappers await `syncCartAction` / `syncWishlistAction` Server Actions (`src/lib/oneentry/auth/actions.ts`).
5. The Server Action reads the `oe_access` cookie, forwards it as Bearer, and calls `getApi().Users.update()` on the user state blob (or the dedicated cart / wishlist endpoints, depending on the tenant configuration).

See `06-ServerActions` for the full inventory and `docs/CART_WISHLIST.md §5` for the debounce semantics.

## Storybook environment

Both API middlewares are still registered in `makeStore()`. In the Storybook decorator (`preview.ts`) `NEXT_PUBLIC_API_URL` is empty, so:

- Direct calls to the query hooks return `{ data: undefined, isError: true }`.
- `useCart()` / `useWishlist()` gate on `isCartApiEnabled()` / `isWishlistApiEnabled()` and simply use the local slice.
- `syncCart` / `syncWishlist` are stubbed out because `AuthContext` never sees a successful `signInAction` result (Storybook has no `ONEENTRY_URL`), so `isLoggedIn` stays `false` and the sync effect is a no-op.

If a story needs to demonstrate Platform-backed behaviour, stub the Server Actions inline — pointing at a real Platform from Storybook is not supported.
