# Redux — RTK Query

There are **5 RTK Query APIs**. Three use `fakeBaseQuery()` + `queryFn` — data
comes from local dataset files via dynamic `import()`; in Storybook these
hooks **work without any mocking**, no network requests are made. The other
two (`cartApi`, `wishlistApi`) use `fetchBaseQuery()` against the real
OneEntry Platform Content API; in Storybook they are auto-disabled because
`NEXT_PUBLIC_API_URL` is empty (`isCartApiEnabled()` / `isWishlistApiEnabled()`
return `false`).

## Handling loading and error states

All RTK Query hooks return `{ data, isLoading, isError, error }`.

**With fallback (recommended in this project):**

```tsx
// Page renders immediately with static data; silently upgrades when query resolves.
const { data: products = WOMEN_CLOTHING_PRODUCTS } = useGetWomenClothingQuery();
```

**With explicit loading/error UI:**

```tsx
const { data: slides, isLoading, isError } = useGetHeroSlidesQuery();

if (isLoading) return <Spinner />;
if (isError)   return <ErrorBanner />;
return <HeroSlider slides={slides!} />;
```

**With skip + conditional rendering:**

```tsx
const { data: recommended, isLoading } = useGetRecommendedQuery(
  product?.recommendedId ?? '',
  { skip: !product?.recommendedId }
);

// isLoading is false when skipped — safe to check
if (!recommended) return null;
return <RecommendedBlock data={recommended} />;
```

> In Storybook, all queries resolve synchronously (fakeBaseQuery + local data).
> `isLoading` will be `false` immediately — no need to handle loading states in stories.

## cartApi · `store/api/cartApi.ts`

Real OneEntry Platform REST under `{NEXT_PUBLIC_API_URL}/users/me/cart`. Uses
`fetchBaseQuery()` with `prepareHeaders` that injects the JWT pulled from
`state.user.data.authToken`. Cache tag: `Cart`.

<table>
  <thead>
    <tr><th>Hook</th><th>Verb</th><th>Notes</th></tr>
  </thead>
  <tbody>
    <tr><td><code>useGetCartQuery()</code></td><td><code>GET</code></td><td>Skip with <code>{ skip: !authToken }</code> — anonymous → 401.</td></tr>
    <tr><td><code>useAddCartItemMutation()</code></td><td><code>POST</code></td><td>Body accepts <strong>absolute</strong> <code>qty</code>, NOT a delta — <code>CartContext</code> computes the resulting total client-side first.</td></tr>
    <tr><td><code>useRemoveCartItemMutation()</code></td><td><code>DELETE</code></td><td>Remove a single line.</td></tr>
    <tr><td><code>useSetCartMutation()</code></td><td><code>PUT</code></td><td>Replace the whole <code>items</code> list. Used by <code>clearCart()</code> with an empty list (no single-call wipe endpoint).</td></tr>
  </tbody>
</table>

`isCartApiEnabled()` returns `false` when `NEXT_PUBLIC_API_URL` is empty —
`CartContext` falls back to the local `cartSlice` only.

## Storybook environment

`cartApi` and `wishlistApi` are imported in `makeStore()` regardless. In the
Storybook decorator (`preview.ts`) no `NEXT_PUBLIC_API_URL` is set, so:

- their hooks return `{ data: undefined, isError: true }` if you call them without `skip`,
- BUT the `CartContext` and `WishlistContext` consult `isCartApiEnabled()` / `isWishlistApiEnabled()` before dispatching anything — so stories that use `useCart()` / `useWishlist()` just work against the local slice.

If you need to demonstrate Platform-backed behaviour in a story, set
`process.env.NEXT_PUBLIC_API_URL` at build time and authenticate via
`AuthContext` first; otherwise prefer the local slice.
