# Redux — Overview

> Stack: **Redux Toolkit** + **RTK Query** · Files: `src/app/store/`

Redux carries **client-side optimistic state only** — cart, wishlist, recently-viewed, catalog UI (filters / sort / view), quick-view / mobile-menu flags, and the user identifier.
**Server-authoritative data** (products, pages, blocks, menus, labels, user profile, orders, reviews, payment accounts) lives in OneEntry and is fetched by **Server Actions + RSC loaders** under `src/lib/oneentry/**` — those results are passed as props into client components; they never enter the Redux store.

---

## Data flow

```
User interaction
      │
      ▼
Context hook  (useCart, useWishlist, useQuickView, useAuth)
      │  calls
      ▼
dispatch(action)                              (optimistic)
      │
      ▼
Redux Reducer  (cartSlice, wishlistSlice, uiSlice …)
      │  updates
      ▼
Redux Store  ──► saveToStorage() ──► localStorage['oe_store']  (v5)
      │
      ▼
useAppSelector / memoised selector
      │
      ▼
Component re-renders
```

**Server sync (signed-in users only):** a debounced 400 ms effect in `CartContext` / `WishlistContext` pushes the Redux snapshot to OneEntry via `syncCartAction` / `syncWishlistAction` Server Actions. On login, `getCurrentUserAction` seeds `AuthContext.user`, and a one-shot sessionStorage-gated merge folds the server snapshot back into Redux. See `docs/CART_WISHLIST.md` and `06-ServerActions`.

**Server-side reads** (products, blocks, pages, labels, menus) do NOT go through Redux — they are awaited inside RSC (`app/**/page.tsx` and `app/layout.tsx`) and threaded down to client components as props.

## Architecture layers

```
┌─────────────────────────────────────────────────────────────┐
│  RSC route shells  (app/**/page.tsx, app/layout.tsx)         │
│     ↕  await loaders in src/lib/oneentry/**                  │
│                                                              │
│  Client components (ProductCard, MiniCart, CatalogTemplate …) │
│     ↕  call context hooks; useAppSelector for reads          │
├─────────────────────────────────────────────────────────────┤
│  Context layer  (CartContext, WishlistContext, AuthContext,  │
│                  QuickViewContext, CatalogAccentContext)     │
│     ↕  thin wrappers — dispatch to slices + call Server Actions
├─────────────────────────────────────────────────────────────┤
│  Redux Store  (6 slices + 2 RTK Query APIs = scaffolding)    │
│     ↕  persisted (cart, wishlist, recentlyViewed, catalog)   │
├─────────────────────────────────────────────────────────────┤
│  OneEntry integration layer  (src/lib/oneentry/**)           │
│  ├── auth/actions.ts       — 18 Server Actions               │
│  ├── activity/actions.ts   — trackActivity                   │
│  ├── blocks/*, catalog/*   — cached loaders (RSC)            │
│  ├── labels/*, menus/*     — cached loaders + React contexts │
│  ├── forms/submit.ts       — submitForm Server Action        │
│  └── payments/accounts.ts  — Payments.getAccounts            │
└─────────────────────────────────────────────────────────────┘
```

**Rule:** For cart, wishlist and quick view — always use context hooks, not dispatch directly.
`catalogSlice` and `userSlice` have no context wrappers — use `useAppDispatch` directly for those.

---

## What is NOT in Redux (intentional)

- Products, pages, blocks, menus, labels, orders, payment accounts — server-fetched via `src/lib/oneentry/**`.
- Auth tokens — httpOnly cookies (`oe_access` / `oe_refresh` / `oe_user`), managed by Server Actions.
- Form state (register, checkout, review, reserve) — local `useState` inside each page/modal.
- Modal open / close for the sign-in / sign-up modal — lives in `AuthContext` React state, not `uiSlice`.

## RootState shape

```ts
{
  cart:           CartState             // src/app/store/cartSlice.ts
  wishlist:       WishlistState         // src/app/store/wishlistSlice.ts
  recentlyViewed: RecentlyViewedState   // src/app/store/recentlyViewedSlice.ts
  catalog:        Record<string, CatalogUIState>  // src/app/store/catalogSlice.ts
  ui:             UIState               // src/app/store/uiSlice.ts
  user:           UserState             // src/app/store/userSlice.ts — authToken/refreshToken kept empty (session in cookies)
  cartApi:        RTK Query cache       // scaffolding — not called by live UI
  wishlistApi:    RTK Query cache       // scaffolding — not called by live UI
}
```

> `cartApi` / `wishlistApi` remain registered in `makeStore()` so any future rollback to a client-side sync path (from Server Actions back to client mutations) would just need `NEXT_PUBLIC_API_URL` and hook calls — no store surgery required.

## Environment

Server-side (required for the SDK to boot):

- `ONEENTRY_URL` — Platform base URL (e.g. `http://localhost:3013`).
- `ONEENTRY_TOKEN` — App token.
- `NEXT_PUBLIC_DEFAULT_LOCALE` — build-time `langCode` (default `en_US`).

Client-side (optional):

- `NEXT_PUBLIC_GOOGLE_CLIENT_ID` — enables the Google Identity Services sign-in button.

`NEXT_PUBLIC_API_URL` from earlier revisions has been **retired** — the SDK now handles all Platform reads/writes server-side.

## Typed hooks

Always use these — never raw `useDispatch` / `useSelector`:

```ts
// From a component inside src/app/
import { useAppDispatch, useAppSelector } from '../store/hooks';

// From a Storybook story
import { useAppDispatch, useAppSelector } from '../app/store/hooks';

const dispatch = useAppDispatch();                        // typed AppDispatch
const items    = useAppSelector(s => s.cart.items);       // typed RootState
```

## Using Redux in components

**Preferred: via context hook** (most components)

```tsx
import { useCart } from '../context/CartContext';

function BuyButton({ product }) {
  const { addItem } = useCart();
  return (
    <button onClick={() => addItem({
      id: product.id,
      name: product.name,
      brand: product.brand ?? '',
      color: '#000000',
      sku: product.id,
      size: 'M',
      quantity: 1,
      price: parseFloat(String(product.price).replace(/[^\d.]/g, '')),
      image: product.image,
    })}>
      Add to Cart
    </button>
  );
}
```

**Direct dispatch** (only when no context wrapper exists — `catalogSlice`, `userSlice`, `uiSlice`)

Actions are exported as **named exports**, not a grouped object:

```tsx
import { useAppDispatch } from '../store/hooks';
import { setSort, toggleFilter } from '../store/catalogSlice';

function SortDropdown({ catalogKey }) {
  const dispatch = useAppDispatch();
  return (
    <select onChange={e => dispatch(setSort({ catalogKey, sortBy: e.target.value }))}>
      …
    </select>
  );
}

// ✅ correct — uiSlice / userSlice also use named exports
import { openQuickView } from '../store/uiSlice';
dispatch(openQuickView({ product, initialColorIndex: null }));

// ❌ wrong — catalogActions does not exist
import { catalogActions } from '../store/catalogSlice';  // undefined
```

## Using Redux in Storybook stories

The global decorator in `preview.ts` already provides all required providers (Redux, AuthProvider, CatalogAccentContext). No manual wrapping needed.

```tsx
// stories/MiniCart.stories.tsx
import React, { useEffect } from 'react';
import { useAppDispatch } from '../app/store/hooks';
import { cartActions } from '../app/store/cartSlice';
import { MOCK_CART_ITEM } from './mockData';
import { MiniCart } from '../app/components/MiniCart';

function WithCartItems() {
  const dispatch = useAppDispatch();
  useEffect(() => {
    dispatch(cartActions.addItem(MOCK_CART_ITEM));
    dispatch(cartActions.openMiniCart());
  }, [dispatch]);
  return <MiniCart />;
}

export const WithItems: Story = { render: () => <WithCartItems /> };
```
