# I18N.md — Internationalization & Localization

> Audit of internationalization (i18n) infrastructure. Audience: LLM agents that need to know whether the project is multi-locale, how copy is sourced, and what changes to add a second language.

---

## 1. Overview — TL;DR

**The project is currently single-locale — `en_US`.** There is no i18n framework, no locale-prefixed routing, no translation dictionaries in the storefront, no language switcher, no RTL handling.

However — **the plumbing is locale-aware end-to-end.** Every OneEntry loader accepts a `lang: Lang = DEFAULT_LOCALE` parameter, `DEFAULT_LOCALE` is configurable via an env var, and OneEntry itself is a multi-locale platform (each entity carries `localizeInfos`). Adding a second locale is a mechanical change:

1. Add `app/[locale]/` route segment.
2. Read `locale` from `await params`.
3. Pass it into every loader as `lang`.
4. Localise UI copy (either through more label markers on the OE side or a per-locale attribute-set variant).

---

## 2. `<html lang>` and metadata locale — the `en-GB` vs `en_US` gap

Two locale strings coexist in the codebase — they refer to different things but the mismatch is worth understanding:

| Constant | Where declared | Purpose |
|---|---|---|
| `en_US` | `src/lib/oneentry/locale.ts::DEFAULT_LOCALE` | `langCode` sent to every OneEntry SDK call — must match a locale that OE has content for. |
| `en-GB` | `app/layout.tsx::<html lang="en-GB">`, `metadata.openGraph.locale`, `metadata.alternates.languages['en-GB']` | Advertised locale for search engines / assistive tech / OG scrapers. |
| `en-GB` | Three `toLocaleDateString('en-GB', …)` calls | Date-format seed. |

The SDK talks to OE in `en_US` because that's the only locale seeded on the current OE tenant, while the outward-facing site advertises `en-GB` because the brand is UK-market. If OE ever adds British-English content, flipping `DEFAULT_LOCALE = 'en_GB'` (and re-seeding attribute sets) will make the two agree.

When multi-locale support lands, both strings should derive from the `locale` param — a small edit that unblocks assistive tech + search-engine locale hints without touching the SDK layer.

## 3. Locale constant

`src/lib/oneentry/locale.ts`:

```ts
export const DEFAULT_LOCALE = (process.env.NEXT_PUBLIC_DEFAULT_LOCALE ?? 'en_US') as 'en_US';
export type Locale = typeof DEFAULT_LOCALE;
```

Every OneEntry SDK call in `src/lib/oneentry/**` accepts `lang: Locale = DEFAULT_LOCALE`. It is passed as `langCode` to the SDK (e.g. `getApi().Products.getProductById(id, {langCode: lang})`).

Overriding at build:

```bash
NEXT_PUBLIC_DEFAULT_LOCALE=de_DE yarn build
```

Would flip the whole storefront to `de_DE` provided the OneEntry tenant has content localised into that locale.

## 4. Where copy comes from

All user-facing strings come from one of four sources, in preference order:

1. **OneEntry AttributesSets (label sets)** — the primary path. `src/lib/oneentry/labels/**` currently ships **12 loader / context pairs**, one per UI surface:

   | Loader (`*-labels.ts`) | React Context (`*LabelsContext.tsx`) | Typed shape (`*-types.ts`) |
   |---|---|---|
   | `account-labels` | `AccountLabelsContext` | `account-types` |
   | `checkout-labels` | `CheckoutLabelsContext` | `checkout-types` |
   | `create-account-labels` | `CreateAccountLabelsContext` | `create-account-types` |
   | `favorites-page-labels` | `FavoritesPageLabelsContext` | `favorites-page-types` |
   | `interface-controls-labels` | `InterfaceControlsLabelsContext` | `interface-controls-types` |
   | `new-arrivals-page-labels` | `NewArrivalsPageLabelsContext` | `new-arrivals-page-types` |
   | `pdp-labels` | `PdpLabelsContext` | `pdp-types` |
   | `product-card-labels` | `ProductCardLabelsContext` | `product-card-types` |
   | `sale-page-labels` | `SalePageLabelsContext` | `sale-page-types` |
   | `sign-in-labels` | `SignInLabelsContext` | `sign-in-types` |
   | `stores-labels` | `StoresLabelsContext` | `stores-types` |
   | `your-bag-labels` | `YourBagLabelsContext` | `your-bag-types` |

   Every loader is a thin wrapper around `getSystemSet(MARKER, lang) → readSystemValue(item, lang)` (see §4.1). Every context follows the identical pattern: `<XLabelsProvider value={…}>` at the layout / page boundary, `useX(key, fallback)` inside components. Fallbacks live in the corresponding `src/app/data/*Labels.ts` static file so components never crash when a marker is missing.

2. **OneEntry system-text primitives** — for eyebrows, section headings, per-page copy that doesn't fit a dedicated label set. See §4.1 below.
3. **Static label constants** in `src/app/data/*Labels.ts` — used as fallbacks when a label context is missing (Storybook stories, SSR cold-start before hydration completes).
4. **Hardcoded English literals** in components — the last resort. Grep periodically to catch drift.

### 4.1 `system-text.ts` primitives

`src/lib/oneentry/system-text.ts` is the low-level layer every label loader sits on. Public surface:

| Export | Kind | Purpose |
|---|---|---|
| `type Lang = 'en_US'` | Type alias | Locked to the single locale seeded on OE. Widens to a union when multi-locale ships. |
| `type SystemSchema` | Type | `Record<string, SystemAttrItem>` — the flat map keyed by attribute identifier. |
| `readSystemValue(item, lang = DEFAULT_LOCALE)` | Function | Extracts a string from either the language-keyed shape (`initialValue.en_US.value`, from REST `/attributes-sets/marker/{m}`) **or** the SDK-flattened shape (`initialValue.value`). Returns `null` on any miss so the caller can decide the fallback. |
| `getSystemSet(marker, lang = DEFAULT_LOCALE)` | Function (`react/cache` + module-level TTL) | Fetches an attribute set by marker with a 5-minute process-wide TTL and in-flight de-duplication. Wraps `AttributesSets.getAttributeSetByMarker`. Returns `{}` on error / disabled SDK so callers stay pure. |
| `t(marker, key, fallback, lang = DEFAULT_LOCALE)` | Function | Convenience wrapper: `getSystemSet` → `readSystemValue` → fallback on empty string / missing key. This is the one-liner used by every loader that only needs a single string. |
| `<SystemText marker key fallback lang?>` | RSC component | Renders `t(marker, key, fallback, lang)` inline. Used for ad-hoc copy that doesn't warrant a dedicated context. |

Vitest coverage in `system-text.test.ts` locks in the five key branches: language-keyed hit, missing key, empty-string value, no schema, SDK throw, and the SDK-disabled short-circuit (no network call when `oneentry` is `null`).

Adding a locale means:

- Making sure attribute sets in OneEntry have localised values.
- The label loaders and system-text already accept `lang`, so no change needed there.
- Rewriting the remaining hardcoded literals (small in number after the migration).

## 5. Currency

`src/app/data/currencyConfig.ts` — currently frozen to USD:

```ts
export const CURRENCY = {
  code: 'USD',
  symbol: '$',
  fmt: (n: number) => `$${n.toFixed(2)}`,
  stripTrailingZeros: (price: string) => price.replace(/\.00$/, ''),
} as const;
```

Consumed by every price render (`ProductCard`, `MiniCart`, `CartPage`, checkout).

**Note the USD-vs-GBP conflict** (see [SEO_OPTIMIZATION.md](./SEO_OPTIMIZATION.md)):

- UI displays USD (`$`).
- JSON-LD structured data (`Product.offers.priceCurrency`, `shippingDetails.shippingRate.currency`) is `GBP` in `src/app/data/seoData.ts`.
- Marketing copy for the London stores implies £ pricing.

This mismatch is intentional-for-now: the storefront is a fashion playground running against a mixed-locale seed dataset. Real currency picking will happen alongside multi-locale routing — likely by adding a currency-per-locale mapping in a new `src/app/data/localeConfig.ts`.

## 6. Locale-aware helpers

- `toLocaleDateString('en-GB', …)` is hardcoded in three places (checkout confirmation, account orders, review dates). Replace with the current locale when multi-locale ships.
- Number formatting via `Intl.NumberFormat` is not currently used — prices go through the `CURRENCY.fmt` helper. When adding locales, wire this to `Intl.NumberFormat(locale, {style: 'currency', currency: 'USD'})`.

## 7. Multi-locale rollout checklist

1. Add `app/[locale]/` segment and colocate every existing route inside.
2. Update `app/layout.tsx` to derive `<html lang>` from `locale`.
3. Add a `LocaleSwitcher` client component; write choice to a `oe_locale` cookie so RSC can read it.
4. Thread `locale` into every RSC loader — most already accept `lang` as first arg after any identifier.
5. Localise `seoData.ts` (`SITE.name` etc.) into a per-locale map or move to OneEntry.
6. Localise currency (see §5).
7. Localise date formatting (see §6).
8. Verify OneEntry has content for the new locale — every product / page / block / attribute-set must be localised or the SDK will fall back to `en_US`.
9. Regenerate `sitemap.ts` to emit `<xhtml:link rel="alternate" hreflang="…">` entries.
10. Regenerate `/llms.txt` per locale.

Nothing structural blocks this — the codebase was written with a single-locale-today / multi-locale-tomorrow shape in mind.

---

## 8. What is intentionally NOT implemented

- No `next-intl`, `react-intl`, `next-i18next` — not needed because copy comes from OneEntry, not a local JSON dictionary.
- No RTL support (no `dir="rtl"` handling in Tailwind config, no `rtl` variants in components).
- No pluralisation library — the only pluralised strings today are hard-coded English.
- No currency conversion — a single price value is served for every render.
- No number/measurement conversion (imperial vs metric).
