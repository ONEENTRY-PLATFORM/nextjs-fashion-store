# I18N.md вЂ” Internationalization & Localization

> Audit of internationalization (i18n) infrastructure in `apps/new-shop-nextjs`. Audience: LLM agents that need to know whether the project is multi-locale, how copy is sourced, and what would need to change to add a second language.

---

## 1. Overview вЂ” TL;DR

**The project is effectively single-locale: English (en-GB).** There is no
i18n framework, no locale-prefixed routing, no translation dictionaries,
no language switcher, no RTL handling, and no Intl-based number formatting.
Every user-facing string is a hard-coded English literal living in a TypeScript
constant file under `src/app/data/*Labels.ts` (or analogous).

The `<html lang="en-GB">` attribute is hard-coded in
`app/layout.tsx:70`, and the OpenGraph metadata declares `locale: 'en_GB'`
(`app/layout.tsx:24`). The sitemap (`app/sitemap.ts`) emits a single URL per
page вЂ” no locale alternates.

**Multi-locale readiness: low.** Adding a second language would require:
1. Introducing an i18n library (`next-intl` / `next-i18next` / custom).
2. Refactoring ~50 `*Labels.ts` / `*Config.ts` files in `src/app/data/` into
   keyed dictionaries with per-locale variants.
3. Adding a `[locale]` URL segment (currently routes live directly under
   `app/`, not `app/[locale]/`).
4. Updating `CURRENCY` in `src/app/data/currencyConfig.ts` to switch by
   locale (currently a frozen `$` / `USD` constant вЂ” note: UK-themed
   content with London stores, but USD pricing).
5. Replacing hard-coded `'en-GB'` arguments in `toLocaleDateString` calls
   with a locale-derived value.
6. Adding a language switcher UI in `Header` / `Footer`.

The OneEntry CMS itself is multilingual-capable (every `localize_infos`
column is a `{<lang>: {...}}` jsonb map вЂ” see global rules around
`oneentry-invariants.md В§7`), but the front-end here does not yet consume
that capability. When the CMS goes live, the SDK's `langCode` argument
will become the natural propagation channel.

## 6. Date / number / currency formatting

### Currency вЂ” frozen single-currency constant

`src/app/data/currencyConfig.ts:1-15` defines `CURRENCY` as:

```ts
export const CURRENCY = {
  symbol: '$',
  code: 'USD',
  format: (n) => `$${n.toFixed(2)}`,
  strip:  (s) => s.replace('$', '').replace(',', ''),
  formatInteger: (n) => `$${n}`,
};
```

The comment explicitly says "When CMS comes online, the symbol/code
switches centrally" вЂ” i.e. currency-per-locale is a future concern. No
`Intl.NumberFormat({ style: 'currency', currency: 'USD' })` is used; the
formatting is a naive `$${n.toFixed(2)}`. `src/app/utils/formatPrice.ts:1-3`
re-exports `CURRENCY.format` as `fmt`.

Note the mismatch: storefront copy and store locations are UK-themed
(London stores in `src/app/data/checkoutConfig.ts:11-15`, postcodes in
labels), but currency is USD with `$`. This is a content inconsistency,
not an i18n feature.

### вљ  Currency authority conflict вЂ” UI says USD, JSON-LD says GBP

There are **two competing currency declarations** in the project, and they
disagree. An LLM (or a future contributor) must know which one is
authoritative.

| Source | Value | Where it appears |
|---|---|---|
| `src/app/data/currencyConfig.ts:6-15` | `'USD'` + `$` symbol + `.toFixed(2)` | **Every UI price** (cart, catalog, product card, checkout totals) via `fmt(n)` from `src/app/utils/formatPrice.ts` |
| `src/app/data/seoData.ts:11` | `'GBP'` | JSON-LD `priceCurrency` on `Product` schema, sitemap shipping-claim templates, OpenGraph price meta |

**Authoritative for the runtime: USD.** Every visible price in the UI is
rendered through `CURRENCY.format` from `currencyConfig.ts`. JSON-LD's `GBP`
is a stale claim вЂ” crawlers reading the structured data will see `GBP ВЈ`
prices that no real page renders. The London-themed store locator and
`DELIVERY_COUNTRY = 'GB'` (`seoData.ts:17`) suggest the original intent was
GBP, but the UI never followed.

**Cleanup direction (when CMS goes live):** unify both files to read from a
single source (probably an `attribute_set.schema.currency` value coming
from OneEntry). Until then, do NOT trust JSON-LD currency / shipping fields
as a description of runtime behaviour вЂ” they are SEO copy only. See
`./SEO_OPTIMIZATION.md` В§"SEO-only claims" for related constants that are
declared but not enforced (`FREE_SHIPPING_THRESHOLD`, `DELIVERY_MIN_DAYS`,
`DELIVERY_MAX_DAYS`, `RETURN_WINDOW_DAYS`).

### Dates вЂ” hard-coded `'en-GB'` arguments

There are three sites that call `toLocaleDateString` with a hard-coded
locale argument:

| File:line | Call |
|---|---|
| `src/app/pages/checkout/DeliveryMethodHome.tsx:185` | `date.toLocaleDateString('en-GB', { weekday: 'short' })` |
| `src/app/pages/checkout/DeliveryMethodHome.tsx:188` | `date.toLocaleDateString('en-GB', { month: 'short' })` |
| `src/app/pages/account/myData/PersonalInfoSection.tsx:102` | `new Date(user.dob).toLocaleDateString('en-GB')` |

When second-locale support arrives, these must take the locale from
context, not the literal `'en-GB'`.

### Number formatting вЂ” locale-agnostic `toLocaleString`

`toLocaleString()` (no argument) is called in:
- `src/app/components/CatalogTemplate.tsx:337,409,438,462,538`
- `src/app/pages/account/LoyaltyCard.tsx:53`
- `src/app/pages/account/BonusesSection.tsx:19`

These produce thousand-separator output in the browser's default locale вЂ”
which differs between users and servers. Not a designed i18n decision,
just a default.

No `Intl.DateTimeFormat` or `Intl.NumberFormat` instances anywhere.

## 8. Default & fallback behaviour

There is no default-fallback chain because there is no translation
lookup. Every string is a direct property access on a constant object
(`ACCOUNT_PAGE_LABELS.pageTitle` в†’ always `'My Account'`). If a label is
missing from the constant, TypeScript surfaces an error at build time
(strict mode). If a runtime string is missing, the component renders
`undefined` вЂ” there is no "fallback to default locale" because there is
no locale dimension.

## 9. SEO вЂ” single-locale hreflang

`app/layout.tsx:45-51`:

```ts
alternates: {
  canonical: SITE_URL,
  languages: {
    'en-GB':     SITE_URL,
    'x-default': SITE_URL,
  },
},
```

Both the language-specific and `x-default` hreflang point to the same
URL вЂ” declaring "this site is `en-GB` only, served at this URL for all
visitors." OG `locale: 'en_GB'` (`app/layout.tsx:24`). Twitter cards
inherit site-wide metadata.

`app/sitemap.ts:6-36` emits one entry per URL with no `alternates` /
`xhtml:link` entries. To add locale alternates you'd extend each entry
with `alternates: { languages: { 'en-GB': url, 'de-DE': url_de } }`.

See `./SEO_OPTIMIZATION.md` for the broader SEO picture.

## Recommendation summary

If multi-locale is on the roadmap:
1. Pick a strategy first вЂ” URL-prefix (`/en/...`, `/de/...`) is the
   industry default and matches Next.js App Router's `[locale]` pattern.
2. Adopt `next-intl` (App-Router-native, RSC-friendly) or `next-i18next`
   (legacy but mature). The current zero-library setup is fine for
   single-locale but does not scale.
3. Migrate `src/app/data/*Labels.ts` to keyed dictionaries
   (`messages/en-GB.json`, `messages/de-DE.json`, вЂ¦). The current
   constant-object pattern is convenient for TypeScript autocompletion;
   preserving it in a typed wrapper around the i18n library is feasible.
4. Replace the hard-coded `'en-GB'` arguments in `toLocaleDateString` and
   the static `CURRENCY` config with locale-aware variants
   (`Intl.DateTimeFormat(locale)`, `Intl.NumberFormat(locale, { style: 'currency', currency })`).
5. Add `hreflang` alternates per locale in `app/layout.tsx` and emit one
   `sitemap.ts` entry per (URL Г— locale) pair.
6. Connect `langCode` to OneEntry SDK calls once the CMS integration
   lands, taking the value from the URL `[locale]` param.

Until then: the project is English-only by design and runs cleanly that way.
